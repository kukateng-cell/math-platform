# K-6 零基礎數學學習平台：MVP 落地計劃

> 本版計劃把「完整 K-6 智慧學習生態系」收斂成可交付的第一版產品。K-6、AI 自適應、B2B 校端、完整 CMS 仍是長期方向，但不放進 MVP 的承諾範圍。

---

## 一、產品定位

第一版只驗證一件事：

**家長能為孩子建立學習檔案，孩子能完成一組低年級數學練習，系統能記錄表現並給出下一步練習建議。**

### MVP 範圍

| 項目 | 第一版決策 |
| --- | --- |
| 目標年級 | K-2 或小一為主，先從「10 以內加減法／數感」開始 |
| 市場模式 | 先做 B2C 家長端；B2B 校端延後 |
| 使用者 | 家長、學生檔案、管理員 |
| 核心流程 | 家長註冊 -> 建立孩子檔案 -> 孩子做題 -> 系統記錄 -> 家長查看 |
| 自適應 | 規則式推薦，不做機器學習 |
| 內容管理 | 簡單 admin 題庫管理，不做完整 CMS |

### 非目標

以下功能不進 MVP：

* BKT／DKT／IRT 等 AI 或心理計量模型
* B2B 學校批次匯入、班級、教師端
* 完整拖曳式知識樹 CMS
* PWA 離線題庫與音檔快取
* Pixi.js 重動畫遊戲化
* 自動 TTS 生成與人工審核管線
* Elasticsearch、Prometheus、Grafana
* 政府證件比對等自建家長同意驗證流程

## 二、核心學習閉環

### 1. 家長建立孩子檔案

家長用 Email 註冊後，可以建立一個或多個孩子檔案。孩子檔案只保存必要資料：

* 暱稱
* 年級或學習階段
* 目前練習單元

孩子第一版不需要獨立帳號與密碼，避免低齡兒童登入流程過重。

### 2. 學生做題

學生進入指定單元後，每次練習 8-12 題。題型先限制在低年級可驗證的範圍：

* 數量比較
* 10 以內加法
* 10 以內減法
* 簡單文字題

互動先用 HTML、CSS、React 元件實作。只有當某個教具必須拖曳或動畫才能說清楚概念時，才加入額外互動。

### 3. 系統記錄表現

每次練習保存：

* 練習開始與完成時間
* 題目、答案、是否正確
* 作答用時
* 是否由家長協助

K-2 可保留「家長協助」按鈕。協助作答會標記 `assisted=true`，不計入能力判斷。

### 4. 下一步推薦

第一版不用「AI」。用簡單、可解釋的規則：

* 最近 5 題錯 3 題：回到前置技能
* 連續答對 5 題：推薦下一個技能
* 正確率 60%-80%：保持當前技能，多練一組
* `assisted=true` 的作答只顯示給家長，不用於推薦

這足以驗證產品體驗，不需要一開始背模型訓練、推論服務與資料科學成本。

## 三、技術方案

### MVP 技術棧

| 領域 | 選型 |
| --- | --- |
| Web 框架 | Next.js + TypeScript |
| 樣式 | Tailwind CSS |
| 資料庫 | PostgreSQL |
| ORM | Prisma |
| 後端 | Next.js Route Handlers / Server Actions |
| 認證 | 家長與管理員登入；學生是家長底下的 profile |
| 部署 | 單一 Web app + 管理式 PostgreSQL |
| 錯誤追蹤 | 上線前接 Sentry；不建自有監控平台 |

### 暫不採用

| 原計劃項目 | 延後原因 |
| --- | --- |
| NestJS 獨立後端 | MVP 邏輯量不值得拆服務 |
| Redis / BullMQ | 暫無大量背景任務 |
| Elasticsearch | 題庫量不足前，資料庫查詢足夠 |
| Prometheus / Grafana | 初期用平台監控與 Sentry 即可 |
| Pixi.js | 先避免低階平板效能風險 |
| PWA 離線 | 離線同步會增加資料一致性與隱私成本 |

## 四、資料模型

第一版只需要以下核心資料：

| 資料表 | 用途 |
| --- | --- |
| `User` | 家長與管理員帳號 |
| `ChildProfile` | 孩子檔案，隸屬於家長 |
| `Skill` | 固定層級的學習技能 |
| `QuestionTemplate` | 題目模板與答案產生邏輯 |
| `PracticeSession` | 一次練習 |
| `Attempt` | 單題作答紀錄 |
| `MasterySnapshot` | 每個孩子在各技能上的簡單掌握狀態 |

技能層級先固定為：

**年級 -> 單元 -> 技能**

不要做無限層級樹。等內容真的複雜到固定三層不夠，再改。

---

## 五、內容與題庫

### 第一版題庫策略

先做少量高品質內容，不追求海量題庫：

* 3-5 個技能
* 每個技能 20-40 題或模板變體
* 每題有答案與簡短解析
* 題目可停用，不直接刪除

### 參數化題目

參數化只支援低風險題型，例如：

* `a + b = ?`
* `a - b = ?`
* `a` 與 `b` 的範圍限制
* 保證答案非負、符合年級範圍

不要一開始做通用公式引擎、富文本公式編輯器或複雜審核流。

### 管理員功能

管理員第一版只需要：

* 新增、修改、停用題目
* 新增、修改技能
* 預覽題目
* 查看最近作答紀錄

完整 CMS 權限分工、審核流程、媒體管理員與數據分析師角色延後。

## 六、使用者介面

### 學生端

第一版頁面：

1. 選擇練習單元
2. 做題頁
3. 完成結果頁

做題頁要求：

* 每次只呈現一個學習焦點
* 支援鍵盤、滑鼠與觸控
* 動畫只用於回饋，不干擾題目
* 提供「減少動畫」模式

### 家長端

第一版頁面：

1. 孩子列表
2. 孩子學習概覽
3. 最近練習紀錄
4. 弱項技能提示

不要做「家長接取任務再派發」這種雙步流程。系統直接推薦今日練習，家長可以接受、略過或更換。

### 管理端

第一版頁面：

1. 技能管理
2. 題目管理
3. 作答紀錄查詢

管理端查閱兒童資料要最小化。預設只顯示暱稱、年級、學習資料；需要排查問題時才查看更多，並留下操作紀錄。

## 七、隱私、合規與安全

本節不是法律意見；正式上線前需要法律顧問確認目標市場要求。

MVP 必做：

* 家長建立孩子檔案前，清楚說明收集哪些資料與用途
* 孩子資料只收集學習必要資訊
* 不接廣告追蹤、不出售資料、不做社交功能
* 家長可以查看、修改、刪除孩子檔案
* 管理員操作敏感資料要留 audit log
* 登入限速，避免暴力破解
* 密碼安全保存

VPC（可驗證家長同意）不要自己硬做。若目標市場要求 COPPA 等兒童隱私合規，優先採用成熟第三方流程，並在公開推出前補齊。

## 八、開發時程

### 第 1-2 週：產品骨架

* 定義第一個學習單元與題型
* 建立 Next.js 專案、資料庫、基本版型
* 完成資料模型初版

### 第 3-4 週：帳號與內容

* 家長註冊與登入
* 孩子檔案
* 管理員登入
* 技能與題目 CRUD

### 第 5-6 週：做題閉環

* 學生做題頁
* 練習結果頁
* 作答紀錄
* 簡單掌握度與下一步推薦

### 第 7-8 週：家長與管理

* 家長學習概覽

* 基本安全檢查
* 錯誤追蹤
* 手機和平板測試
* 少量家庭內測

### 第 11-12 週：修正與 beta

* 根據內測修正題目、流程與 UI
* 補足隱私說明與刪除流程
* 開放小規模 beta

## 九、上線判斷標準

不要用「功能做完」判斷成功。第一版只看以下指標：

| 指標 | 目標 |
| --- | --- |
| 首次練習完成率 | >= 70% |
| 平均每次練習完成題數 | >= 8 題 |
| 家長能看懂學習紀錄 | 內測訪談中多數能說出孩子弱項 |
| 題目錯誤率異常 | 可由管理員快速停用問題題目 |
| 平板體驗 | 主流低階 Android 平板可順暢完成一輪練習 |

達不到這些指標時，不進入 AI、B2B 或大型 CMS。

## 十、MVP 後路線圖

只有在 MVP 數據成立後，才進入下一階段：

1. 擴展到更多 K-2 單元
2. 加入更多互動教具
3. 加入雲端 TTS 或錄音素材
4. 做輕量內容審核流
5. 嘗試 B2B 小型學校試點
6. 累積足夠作答資料後，再評估 BKT、IRT 或其他模型

原則：先證明孩子願意做、家長看得懂、內容能維護，再談完整平台。
